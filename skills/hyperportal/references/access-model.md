# Access model

## Capability links, per org

Access is a **team-level grant**: one link per counterparty organization,
shared internally by them. `/r/<token>` validates against `PORTAL_TEAMS`, sets
an HttpOnly cookie (`dp_access`), and redirects to `/room`. The token leaves
the address bar immediately — it never lands in browser history beyond the
first hit, a referrer header, or a screenshot of the room.

Why per-org, not per-person: the org is the unit that always exists. Requiring
individuals to register before reading is friction that loses reads; the
counterparty forwards the link internally anyway. Individual identity is
collected as a **courtesy** — an optional modal (`dp_whoami` cookie, PostHog
`identify`) that is skippable and never re-asked after dismissal.

## The Team record

```ts
type Team = {
  token: string;      // 32-hex, generated by the CLI only
  org: string;        // counterparty name — footer + PostHog group key
  active: boolean;    // false = revoked (kept for history; --delete removes)
  tier?: number;      // disclosure tier; absent/invalid ⇒ 1
  expires?: string;   // ISO date; evaluated per request (no redeploy needed)
};
```

Stored as JSON in the `PORTAL_TEAMS` env var. Extra fields round-trip through
the CLI untouched, so adding a field needs no serialization change.

## Fail-closed, everywhere

- Malformed `PORTAL_TEAMS` JSON → `parseTeams()` returns `[]` → **everyone**
  locked out. This is deliberate: a parse bug must never widen access. (It's
  also why only the CLI, which serialises via `JSON.stringify`, may write it.)
- `tierOf(team)`: anything but a valid integer ≥ 1 reads as tier 1 — access
  widening is always explicit.
- Unknown/inactive/expired token → 403 at the gate, locked page at `/room`.

## Tiers

Recipient `tier` vs section `minTier`, with `belowTier` controlling the
below-tier rendering:

- `"teaser"` — greyed "on request" row in the index. The recipient knows the
  section exists and can ask. Use when advertising the material is useful.
- `"hidden"` — omitted entirely: not in the index, not in the nav, and the
  URL 404s **indistinguishably from a nonexistent page**. Use when the
  section's existence would itself plant an unwanted question.

No per-document tiers: a document inherits its section's `minTier`. If one
document needs a different stage, that's a signal it belongs in a different
section.

## Server-side enforcement: three places, always

1. **Room index** (`app/room/page.tsx`) — link / teaser pill / omitted.
2. **Section page** (`app/room/[slug]/page.tsx`) — `notFound()` unless
   `canView(section, tier)`. Prev/next navigation walks only viewable sections.
3. **Document stream** (`app/room/docs/[id]/route.ts`) — resolves the doc's
   section and re-checks `canView`; 404 otherwise.

When you add any new content path (an API route, a search endpoint, an RSS
feed…), it re-checks the cookie and tier itself. A UI that merely doesn't
render a link is not a gate.

## What this model does NOT give you

- Per-person access control or audit-grade identity (identification is
  voluntary). If a portal legally requires signed individual access, this
  model is the wrong base.
- Instant revocation — see operations.md (revoke = redeploy latency).
- Watermarking / download prevention. Assume anything disclosed can be saved.
